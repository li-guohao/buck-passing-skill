# 甩锅案例集 / Buck-Passing Examples

## 案例1: 项目延期甩锅 / Project Delay

### 背景 / Background
- **场景**: 季度项目延期2周交付
- **涉及方**: 开发团队、产品经理、项目经理
- **会议**: 延期复盘会

### 用户输入 / User Input
> 项目延期了，老板在群里问怎么回事，产品说需求早就给了。

### Skill 回应 / Skill Response

```
【局势分析 / Situation Analysis】

这是一个典型的多方责任场景。产品说"需求早给了"，
但可能忽略了后续的需求变更。

推荐策略: 温和澄清型 (Level 2)

【推荐话术 / Suggested Response】

版本A - 温和型:
"这个项目确实延期了，作为技术负责人我负主要责任。
回顾整个过程，我们在V1版本评估后，后续有3次需求变更，
每次我都做了时间影响评估并同步给了产品同学。
建议我们一起看看如何优化变更流程，避免类似情况。"

版本B - 证据型:
"延期确实发生了，我整理了一下时间线：
- 3月1日: 需求V1，评估工期2周
- 3月5日: 需求变更V2，+3天
- 3月10日: 需求变更V3，+5天
- 3月12日: 新增功能点，+4天
原始需求确实早给了，但变更累积导致了延期。
"

版本C - 团队型:
"这个问题我们团队需要一起复盘。从开发侧看，
我们在每个变更点都及时响应了，但可能缺少整体的变更管理。
建议建立变更影响评估机制，技术、产品、项目三方确认后再执行。"
```

## 案例2: 线上故障甩锅 / Production Incident

### 背景 / Background
- **场景**: 线上系统崩溃2小时
- **涉及方**: 开发、运维、测试、DBA
- **会议**: 故障复盘会

### 用户输入 / User Input
> 系统昨晚挂了，运维说是代码问题，但明明测试都通过了。

### Skill 回应 / Skill Response

```
【局势分析 / Situation Analysis】

这是一个高压场景，各方都在推卸责任。
关键是找到root cause，而不是互相指责。

推荐策略: 系统分析型 (Level 3)

【推荐话术 / Suggested Response】

版本A - 技术型:
"关于昨晚的故障，我已经做了初步分析。
从日志看，故障触发点是数据库连接池耗尽。
这个配置在测试环境和生产环境不一致，
测试环境max_connections=100，生产是50。
建议我们统一环境配置标准，避免类似问题。"

版本B - 流程型:
"这次故障暴露了我们发布流程的问题。
代码在测试环境验证通过，但生产环境配置不同。
这不是某个人的问题，而是环境管理的问题。
建议引入配置管理工具，确保环境一致性。"

版本C - 担当型（反向操作）:
"作为开发负责人，这个故障我负主要责任。
虽然测试通过了，但我没有考虑到生产环境的差异。
我已经整理了改进方案，包括配置检查和监控告警，
请大家帮忙review一下。"
```

## 案例3: 需求理解错误甩锅 / Misunderstood Requirements

### 背景 / Background
- **场景**: 开发的功能和产品预期不一致
- **涉及方**: 开发、产品经理
- **会议**: 需求评审会

### 用户输入 / User Input
> 我做完了功能，产品说不是他想要的，但PRD里确实这么写的啊。

### Skill 回应 / Skill Response

```
【局势分析 / Situation Analysis】

这是一个典型的"期望鸿沟"问题。
PRD写的是A，产品想的是B，你做的是A。

推荐策略: 文档对齐型 (Level 2)

【推荐话术 / Suggested Response】

版本A - 对齐型:
"我按照V2版本的PRD实现了功能A。
如果和预期有偏差，可能是PRD没有完整表达产品意图。
建议我们一起更新PRD，确保文档和期望一致，
我这边可以快速调整。"

版本B - 学习型:
"这个功能我确实是按照PRD实现的，
但可能我对某些细节的理解和产品同学有偏差。
能否麻烦产品同学指出具体哪里不符合预期？
我记录下来，以后避免类似问题。"

版本C - 流程型:
"这次偏差说明我们的需求确认流程可以优化。
建议在PRD评审后，增加原型确认环节，
确保开发和产品的理解一致再进入开发。
这次的功能我可以快速调整。"
```

## 案例4: 跨部门协作甩锅 / Cross-department Issues

### 背景 / Background
- **场景**: 项目延期，涉及多个部门
- **涉及方**: 前端、后端、设计、产品
- **会议**: 项目延期分析会

### 用户输入 / User Input
> 后端说前端延期导致整体延期，但前端说接口给晚了。

### Skill 回应 / Skill Response

```
【局势分析 / Situation Analysis】

这是典型的"踢皮球"场景，各方都有道理。
需要上升到流程和协作层面，而不是纠结谁对谁错。

推荐策略: 协作优化型 (Level 2)

【推荐话术 / Suggested Response】

如果你是后端:
"从时间线看，我们在约定时间提供了接口文档V1，
后续有3个字段调整，都在群里同步了。
建议我们建立接口变更通知机制，
确保前端同学能及时收到更新。"

如果你是前端:
"接口V1确实按时提供了，但开发过程中有几次字段变更，
导致需要返工。建议我们在接口冻结后再进入开发，
或者建立变更影响评估机制。"

如果你是项目经理:
"这个问题反映了我们在跨部门协作上的改进空间。
建议建立:
1. 接口冻结机制
2. 变更影响评估流程
3. 每日站会同步阻塞点
这次延期大家都有责任，我们一起优化流程。"
```

## 案例5: 上级决策甩锅 / Superior Decision Issues

### 背景 / Background
- **场景**: 按照领导要求做的方案出了问题
- **涉及方**: 执行者、决策者
- **会议**: 问题复盘会

### 用户输入 / User Input
> 这个方案是老板拍板的，现在出问题了，老板在问怎么回事。

### Skill 回应 / Skill Response

```
【局势分析 / Situation Analysis】

这是高难度场景，需要给老板留面子，同时保护自己。
关键是强调"当时的信息"和"现在的认知"差异。

推荐策略: 经验总结型 (Level 3)

【推荐话术 / Suggested Response】

版本A - 学习型:
"当时的决策是基于当时的市场信息和数据做出的，
这个决策在当时看是合理的。
现在市场情况发生了变化，我们获得了新的信息，
建议基于新信息调整策略。
这次的经验是，我们需要建立更灵活的策略调整机制。"

版本B - 前瞻型:
"这个方案在执行过程中遇到了一些预期外的情况，
包括[具体情况]。
我们已经在调整方案，预计[时间]可以恢复正常。
建议以后在方案设计时增加风险评估和应急预案。"

版本C - 担当型（慎用）:
"作为执行负责人，我没有及时识别风险并上报，
这是我的失误。
当时老板决策时我应该在执行层面提出更多风险点。
现在已经制定了补救措施，包括[措施]。"
```
